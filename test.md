# 代码库执行计划 (ExecPlans)：

本文档描述了对执行计划（"ExecPlan"）的要求，这是一份编码代理可以遵循的设计文档，用以交付一个可工作的功能或系统变更。请将读者视为本代码库的完全初学者：他们只有当前的工作目录树和你提供的单个 ExecPlan 文件。没有对先前计划的记忆，也没有外部上下文。

## 如何使用 ExecPlans 和 PLANS.md

在编写可执行规范（ExecPlan）时，请**严格**遵循 PLANS.md 文件。如果它不在你的上下文中，请通过阅读整个 PLANS.md 文件来刷新你的记忆。仔细阅读（并重新阅读）源材料，以产生准确的规范。在创建规范时，从骨架开始，并在研究过程中逐步充实。

在实现一个可执行规范（ExecPlan）时，不要提示用户"下一步"；只需继续推进到下一个里程碑。保持所有部分最新，在每个停止点添加或拆分列表条目，以明确陈述已取得的进展和后续步骤。自主解决歧义，并频繁提交。

在讨论一个可执行规范（ExecPlan）时，在规范的日志中记录决策以备将来参考；任何对规范的修改原因都应该是清晰无误的。ExecPlans 是动态文档，应该始终能够**仅**从 ExecPlan 重新开始，而不依赖任何其他工作。

在研究具有挑战性要求或重大未知因素的设计时，使用里程碑来实现概念验证、"玩具实现"等，以验证用户的提议是否可行。通过查找或获取库的源代码进行阅读，深入研究，并包含原型以指导更完整的实现。

## 要求

**非协商性要求**：

*   每个 ExecPlan 必须是完全自包含的。自包含意味着以其当前形式，它包含了新手成功所需的所有知识和说明。
*   每个 ExecPlan 都是一个动态文档。贡献者需要在取得进展、有新发现以及设计决策最终确定时修订它。每次修订后，文档必须保持完全自包含。
*   每个 ExecPlan 必须使一个完全的新手能够在没有本代码库先验知识的情况下，端到端地实现该功能。
*   每个 ExecPlan 必须产生一个可证明有效的行为，而不仅仅是"满足定义"的代码变更。
*   每个 ExecPlan 必须用通俗语言定义每一个专业术语，否则就不要使用它。

目的和意图优先。首先用几句话解释，从用户的角度来看，这项工作为何重要：变更后，人们能够做哪些之前无法做到的事情，以及如何看到它的效果。然后引导读者完成实现该结果的确切步骤，包括编辑什么、运行什么以及他们应该观察到什么。

执行你的计划的代理可以列出文件、读取文件、搜索、运行项目和运行测试。它不知道任何先前的上下文，也无法从早期的里程碑推断你的意图。重复你依赖的任何假设。不要指向外部的博客或文档；如果需要知识，请将其用自己的话嵌入计划本身。如果一个 ExecPlan 建立在先前的 ExecPlan 之上，并且该文件已签入，则通过引用来合并它。如果没有，你必须包含该计划中的所有相关上下文。

## 格式

格式和封装是简单且严格的。每个 ExecPlan 必须是单个用 `md` 标签标识的围栏代码块，以三个反引号开始和结束。不要在内部嵌套额外的三反引号围栏代码块；当你需要展示命令、日志、差异或代码时，将它们作为该单个围栏内的缩进块呈现。为了清晰起见，请使用缩进，而不是在 ExecPlan 内部使用代码围栏，以避免提前结束 ExecPlan 的代码围栏。在每个标题后使用两个换行符，使用 # 和 ## 等，并对有序和无序列表使用正确的语法。

当将 ExecPlan 写入一个 Markdown (.md) 文件，且该文件的内容***仅***包含该单个 ExecPlan 时，你应该省略三个反引号。

用平实的散文写作。优先使用句子而不是列表。避免使用清单、表格和长篇枚举，除非简洁会掩盖含义。清单仅允许在 `进度` 部分使用，并且在那里是强制性的。叙述性部分必须保持以散文为主。

## 指导原则

自包含性和通俗语言至关重要。如果你引入一个非普通英语的短语（"守护进程"、"中间件"、"RPC网关"、"过滤器图"），请立即定义它，并提醒读者它在本代码库中是如何体现的（例如，通过命名它出现的文件或命令）。不要说"如先前定义"或"根据架构文档"。在这里包含所需的解释，即使是在重复自己。

避免常见的失败模式。不要依赖未定义的术语。不要将"功能的字面要求"描述得过于狭隘，以至于产生的代码能编译但无实际意义。不要将关键决策外包给读者。当存在歧义时，在计划本身中解决它，并解释你选择该路径的原因。宁可过度解释用户可见的效果，也不要过度规定非核心的实现细节。

用可观察的结果来锚定计划。说明用户在实现后可以做什么，要运行的命令，以及他们应该看到的输出。验收标准应表述为人类可以验证的行为（"启动服务器后，导航到 [http://localhost:8080/health](http://localhost:8080/health) 返回 HTTP 200 且响应体为 OK"），而不是内部属性（"添加了 HealthCheck 结构体"）。如果变更涉及内部修改，请解释其影响如何仍然可以被演示（例如，通过运行在变更前失败、变更后通过的测试，并展示使用新行为的场景）。

明确指定代码库上下文。使用完整的相对于代码库根目录的路径命名文件，精确命名函数和模块，并描述应在何处创建新文件。如果涉及多个区域，请包含一个简短的定位段落，解释这些部分如何组合在一起，以便新手可以自信地进行导航。在运行命令时，显示工作目录和确切的命令行。当结果取决于环境时，陈述假设并在合理时提供替代方案。

保持幂等性和安全性。编写的步骤应该可以运行多次而不会造成损害或偏差。如果某个步骤可能中途失败，请包含如何重试或调整。如果迁移或破坏性操作是必要的，请详细说明备份或安全回退方案。优先采用可加性的、可测试的变更，以便在过程中进行验证。

验证不是可选的。包含运行测试的说明，如果适用则启动系统，并观察它做一些有用的事情。描述对任何新功能或能力的全面测试。包含预期的输出和错误消息，以便新手能够区分成功与失败。在可能的情况下，展示如何证明变更在编译之外也是有效的（例如，通过一个小型的端到端场景、CLI 调用或 HTTP 请求/响应日志）。说明适合项目工具链的确切测试命令，以及如何解释其结果。

捕获证据。当你的步骤产生终端输出、简短的差异或日志时，将它们作为缩进示例包含在单个围栏块内。保持简洁并专注于证明成功的部分。如果需要包含补丁，优先使用文件范围的差异或小型摘录，读者可以通过遵循你的指令重新创建它们，而不是粘贴大块内容。

## 里程碑

里程碑是叙述性的，而不是官僚主义的。如果你将工作分解为里程碑，请用简要的段落介绍每个里程碑，描述其范围、里程碑结束时将存在哪些之前没有的东西、要运行的命令以及你期望观察到的验收标准。保持其作为故事的可读性：目标、工作、结果、证明。进度和里程碑是不同的：里程碑讲述故事，进度跟踪细粒度的工作。两者都必须存在。绝不要为了简洁而缩写里程碑，也不要省略对未来实现可能至关重要的细节。

每个里程碑必须是独立可验证的，并逐步实现执行计划的整体目标。

## 动态计划和设计决策

*   ExecPlans 是动态文档。当你做出关键设计决策时，更新计划以记录决策及其背后的思考。将所有决策记录在 `决策日志` 部分。
*   ExecPlans 必须包含并维护 `进度`、`惊喜与发现`、`决策日志` 和 `结果与回顾` 部分。这些不是可选的。
*   当你发现优化器行为、性能权衡、意外错误或影响你方法方向的逆变换/逆应用语义时，请将这些观察捕获到 `惊喜与发现` 部分，并附上简短的证据片段（测试输出是理想的选择）。
*   如果你在实施过程中改变路线，请在 `决策日志` 中记录原因，并在 `进度` 中反映其影响。计划既是你自己的清单，也是为下一位贡献者提供的指南。
*   在完成一项主要任务或整个计划时，写一个 `结果与回顾` 条目，总结已实现的成果、剩余工作以及经验教训。

# 原型制作里程碑和平行实现

在它们能够降低更大变更的风险时，包含明确的原型制作里程碑是可以接受的——而且通常是值得鼓励的。例如：向依赖项添加低层操作符以验证可行性，或者在衡量优化器效果的同时探索两种组合顺序。保持原型的可加性和可测试性。明确地将范围标记为"原型制作"；描述如何运行和观察结果；并说明推广或丢弃原型的标准。

优先采用可加性的代码变更，然后进行删除操作，同时保持测试通过。平行实现（例如，在迁移期间将适配器与旧路径一起保留）在降低风险或使测试在大型迁移期间继续通过时是可取的。描述如何验证两条路径，以及如何通过测试安全地停用其中一条。当处理多个新库或功能领域时，考虑创建独立的探针来评估这些功能**相互独立**的可行性，证明外部库的表现符合预期，并在隔离状态下实现了我们需要的功能。

## 一个好的 ExecPlan 的骨架

    # <简短、面向操作的描述>

    此 ExecPlan 是一个动态文档。`进度`、`惊喜与发现`、`决策日志` 和 `结果与回顾` 部分必须在工作推进过程中保持最新。

    如果 PLANS.md 文件已签入代码库，请在此处从代码库根目录引用该文件的路径，并注明本文档必须根据 PLANS.md 进行维护。

    ## 目的 / 整体概览

    用几句话解释某人在此变更后能获得什么，以及他们如何看到其效果。说明你将实现的用户可见行为。

    ## 进度

    使用带复选框的列表来总结细粒度步骤。每个停止点都必须记录在这里，即使需要将一个部分完成的任务拆分为两个（"已完成"与"剩余"）。此部分必须始终反映工作的实际当前状态。

    - [x] (2025-10-01 13:00Z) 示例完成的步骤。
    - [ ] 示例未完成的步骤。
    - [ ] 示例部分完成的步骤（已完成：X；剩余：Y）。

    使用时间戳来衡量进展速度。

    ## 惊喜与发现

    记录在实施过程中发现的意外行为、错误、优化或见解。提供简洁的证据。

    - 观察：……
      证据：……

    ## 决策日志

    记录在处理计划时做出的每个决策，格式如下：

    - 决策：……
      理由：……
      日期/作者：……

    ## 结果与回顾

    在主要里程碑或完成时总结成果、差距和经验教训。将结果与最初的目的进行比较。

    ## 上下文与定位

    描述与本任务相关的当前状态，假设读者一无所知。通过完整路径命名关键文件和模块。定义你将使用的任何不明显的术语。不要提及先前的计划。

    ## 工作计划

    用散文描述编辑和添加的顺序。对于每次编辑，命名文件和位置（函数、模块）以及要插入或更改的内容。保持具体和简洁。

    ## 具体步骤

    说明要运行的确切命令以及在哪里运行它们（工作目录）。当命令生成输出时，显示一个简短的预期日志，以便读者进行比较。此部分必须在工作推进过程中更新。

    ## 验证与验收

    描述如何启动或操作系统以及观察什么。将验收标准表述为行为，包括具体的输入和输出。如果涉及测试，应说"运行 <项目的测试命令> 并预期 <N> 通过；新测试 <名称> 在变更前失败，在变更后通过"。

    ## 幂等性与恢复

    如果步骤可以安全地重复，请说明。如果步骤有风险，请提供安全的重试或回滚路径。完成后保持环境清洁。

    ## 工件与备注

    将最重要的日志、差异或代码片段作为缩进示例包含在内。保持简洁并专注于证明成功的部分。

    ## 接口与依赖项

    要有规定性。命名要使用的库、模块和服务及其原因。指定在里程碑结束时必须存在的类型、特质/接口和函数签名。优先使用稳定的名称和路径，例如 `crate::module::function` 或 `package.submodule.Interface`。例如：

    在 crates/foo/planner.rs 中，定义：

        pub trait Planner {
            fn plan(&self, observed: &Observed) -> Vec<Action>;
        }

如果你遵循上述指导原则，一个单一的、无状态的代理——或者一个人类新手——就可以从头到尾阅读你的 ExecPlan，并产生一个可工作的、可观察的结果。这就是标准：**自包含、自给自足、新手引导、以结果为中心**。

当你修订一个计划时，你必须确保你的更改在所有部分（包括动态文档部分）得到全面反映，并且你必须在计划底部写一个说明，描述更改及其原因。ExecPlans 不仅必须描述"是什么"，还必须描述几乎每件事情的"为什么"。
