# Coding Discipline

## 1 命名

- 名字即文档，无需注释辅助。
- `snake_case` 用于变量和函数，`CamelCase` 仅用于类。
- 函数名用动词：`collect_rent`。布尔值用 `is_`/`has_`/`can_` 前缀。
- 不缩写（`mgr`→`manager`）。作用域越大名字越具体；循环变量可短（`i`、`k, v`）。

## 2 函数

- 一个函数只做一件事。判断标准：能否不用"和"字描述它。
- 函数体内语句保持同一抽象层级。高层编排步骤，低层执行细节，不混在一起。
- 参数 ≤ 3 个，超过收进 table。不用 flag 参数切换行为，拆成两个函数。
- 前置校验，提前返回，主路径在最外层。
- 查询不改状态，命令不返回数据（操作结果除外）。
- 优先用 `local function name()` 而非 `local name = function()`。

## 3 文件与模块

- 一个文件一个修改理由。
- 文件长度是气味，不是红线。拆分的收益必须大于新增文件带来的导航和协调成本。
- 所有 `require` 放文件顶部。所有非导出符号用 `local`。
- 模块暴露最小接口；依赖单向，低层通过回调或注入与高层通信。

## 4 格式

- 文件按报纸版式：公开接口在前，细节在后，从上往下读即可理解。
- 相关代码物理距离近；垂直留白分隔概念，不做装饰。
- 已知属性用 `.`，动态键用 `[]`。table 字面量末尾加逗号。

## 5 错误处理

- 明确处理或传播，不吞错误。失败返回 `nil, err_msg`。
- 错误信息含上下文：发生了什么、在哪里、为什么。

## 6 注释与文档

- 删除解释"做了什么"的注释，只留解释"为什么"。
- 禁止注释掉的代码。TODO 必须关联具体任务。

## 7 数据结构

- 默认 table + function。不用 metatable 除非显著简化。
- 字段名定义一次、引用多次，不硬编码散落。

## 8 测试

- 改过的逻辑必须有验证手段。
- 一个测试验证一个行为，测试名说清条件、动作、预期。

## 9 简化优于添加

- 第一反应：能不能删代码。未使用的符号直接删，不注释保留。
- 只做转发的包装层是噪音，删掉。
- 重复不自动等于坏。抽象前先问：这两处会因同一原因一起变吗？恰好相似但语义独立的代码，各自保留比强行合并更安全。

## 10 变更纪律

- 只改需要改的。重构与功能变更不在同一次提交。
- 改完后代码比改前更干净——底线，不是额外要求。
